# Wallet 项目宪章

## 核心原则

### I. 简洁优先

- 技术选型必须（MUST）优先选择最简单、最成熟的方案
- 禁止（MUST NOT）引入仅为"未来可能用到"的框架或抽象层
- 代码结构必须（MUST）保持扁平，避免超过三层的目录嵌套
- 每个模块的职责必须（MUST）一句话能说清楚

**理由**: 简单的代码更容易理解、调试和维护。复杂度是项目失败的
首要原因。

### II. 快速交付

- 每个功能必须（MUST）以最小可用产品（MVP）为目标先行交付
- 必须（MUST）优先实现核心用户路径，延后处理边缘场景
- 单个任务的实现范围必须（MUST）控制在可独立验证的最小单元
- 禁止（MUST NOT）在未交付 MVP 前进行代码重构或优化

**理由**: 尽早交付可工作的软件，通过真实使用反馈指导后续迭代，
避免过度设计。

### III. 中文文档

- 所有项目文档必须（MUST）使用中文撰写
- 代码注释应当（SHOULD）使用中文，除非引用外部 API 术语
- 变量名和函数名必须（MUST）使用英文，遵循所用语言的命名惯例
- Git 提交信息应当（SHOULD）使用中文

**理由**: 中文文档降低团队沟通成本，提升文档的可读性和维护意愿。

### IV. 务实测试

- 核心业务逻辑必须（MUST）有测试覆盖
- 测试应当（SHOULD）聚焦于用户可感知的行为，而非实现细节
- 禁止（MUST NOT）为了追求覆盖率而编写无意义的测试
- 集成测试优先于单元测试，除非单元测试能显著加速开发反馈循环

**理由**: 测试是保障质量的手段而非目的。务实的测试策略在质量和
开发速度之间取得平衡。

### V. 最小依赖

- 每个外部依赖的引入必须（MUST）有明确的理由
- 能用标准库解决的问题禁止（MUST NOT）引入第三方包
- 依赖数量必须（MUST）定期审查，移除不再使用的依赖
- 优先选择零依赖或依赖树浅的库

**理由**: 依赖越少，供应链风险越低，构建越快，升级越简单。

## 技术栈约束

- 技术选型必须（MUST）优先考虑团队已有经验的技术
- 新技术引入必须（MUST）经过原型验证，证明其收益大于学习成本
- 构建和部署流程必须（MUST）能在单条命令内完成
- 开发环境搭建必须（MUST）能在 10 分钟内完成
- 禁止（MUST NOT）同时引入超过一个团队不熟悉的新技术

## 开发流程

- 功能开发必须（MUST）遵循: 规格说明 → 计划 → 任务 → 实现 的流程
- 每个功能分支必须（MUST）从 main 分支创建
- 代码提交必须（MUST）保持原子性，每次提交解决一个问题
- 遇到需求不明确时必须（MUST）先澄清再动手编码
- 应当（SHOULD）每天至少提交一次可工作的代码

## 治理

- 本宪章是项目的最高指导原则，所有开发决策必须（MUST）遵守
- 宪章修订必须（MUST）记录修改内容、理由和影响范围
- 宪章版本遵循语义化版本规则:
  - MAJOR: 原则删除或根本性重定义
  - MINOR: 新增原则或实质性扩展
  - PATCH: 措辞修正、笔误修复
- 每次功能规划时应当（SHOULD）检查宪章合规性
